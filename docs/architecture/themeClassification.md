# Theme Classification Architecture — Google Play Review Intelligence (Module A)

> Resolves: ADR-003 (Theme clustering approach)
> Depends on: ADR-002 (Embedding model — Gemini text-embedding-004), ADR-009 (LLM model-per-task), ADR-010 (LLM call caching)
> Model note: Per ADR-009, the project uses Gemini 2.5 Flash for generation and Flash-Lite for classification/safety. Gemini 2.0 Flash is retired.
> Cost target: **$0** — every component must be free tier or fully local.

---

## 1. Why ML Clustering Over Keyword Matching

Keyword matching (e.g., regex rules like `login|OTP|password → "Login Issues"`) fails for review intelligence because:

1. **Synonymy.** Customers describe the same problem in many ways: "can't log in," "OTP never arrives," "authentication broken," "keeps signing me out." A keyword list would need to enumerate every variation and still miss novel phrasing.

2. **Polysemy.** The word "update" appears in reviews about nominee updates, app updates, KYC updates, and statement updates. Keyword matching assigns all to the same bucket. Embedding-based clustering separates them by semantic context.

3. **Emerging themes.** Keyword rules only catch themes the developer anticipated. When a new issue surfaces (e.g., "dark mode broken after update"), keyword rules miss it entirely. ML clustering discovers new groupings from the data without prior configuration.

4. **Maintenance burden.** A keyword dictionary requires manual updates as product features and user language evolve. Clustering adapts automatically each week.

5. **Theme ranking.** Keyword matching counts string hits; it cannot rank themes by semantic density or distinguish a dominant complaint from a niche one. Clustering naturally produces cluster sizes that serve as a ranking signal.

ML clustering with embeddings captures semantic meaning, handles synonyms, discovers emergent issues, and requires zero manual keyword maintenance.

## 2. Clustering Algorithm Evaluation

All candidates must satisfy: **free, runs locally, no GPU required.**

### 2.1 K-Means

**How it works:** Partitions N data points into k clusters by minimizing within-cluster sum of squared distances to centroids. Requires k as an input parameter.

**If selected, the following would be needed:**

- **Elbow method:** Run K-Means for k = 2 to 10, plot inertia (within-cluster sum of squares) vs k, select the "elbow" point where marginal improvement drops.
- **Silhouette score:** For each k, compute the mean silhouette coefficient (range -1 to 1; higher = better-separated clusters). Pick the k with the highest silhouette score.
- **Label generation:** Cluster centroids are 768-dim vectors with no human-readable meaning. Labels must be generated by sending representative reviews from each cluster to an LLM.

**Strengths:** Fastest algorithm, deterministic (with fixed seed), available in TypeScript (`ml-kmeans`).

**Weaknesses:** Must choose k upfront (elbow/silhouette adds complexity), assumes spherical clusters, assigns every review to a cluster (no noise handling — generic or irrelevant reviews pollute themes), no auto-labeling.

### 2.2 HDBSCAN

**How it works:** Hierarchical Density-Based Spatial Clustering. Finds clusters of varying density without requiring k. Points in low-density regions are labeled as noise (cluster = -1).

**Strengths:** Automatically discovers the number of clusters, handles noise (irrelevant or generic reviews get label -1 and are excluded from themes), works well with non-spherical cluster shapes, robust to outliers.

**Weaknesses:** No built-in readable labels (still needs LLM for label generation), slower than K-Means on large datasets (negligible at ~180 reviews), requires dimensionality reduction (UMAP) for best results on high-dimensional embeddings, Python-only (no quality TypeScript implementation).

### 2.3 BERTopic (UMAP + HDBSCAN + c-TF-IDF)

**How it works:** A three-stage pipeline:
1. **UMAP** reduces embedding dimensions (768 → 5) to improve clustering quality.
2. **HDBSCAN** discovers clusters with automatic k and noise handling.
3. **c-TF-IDF** (class-based TF-IDF) computes the most representative terms per cluster, producing human-readable keyword labels without any LLM call.

The project uses pre-computed Gemini text-embedding-004 embeddings (768-dim), so BERTopic's internal embedding step is skipped entirely — no sentence-transformers or PyTorch dependency.

**Strengths:** Auto k, noise handling, AND readable c-TF-IDF labels in a single pipeline. Labels can be optionally refined with a cheap LLM call but are already interpretable without one. Actively maintained, well-documented.

**Weaknesses:** Python-only, heavier dependency footprint than K-Means (though lighter than it appears when using pre-computed embeddings). Requires UMAP + HDBSCAN + scikit-learn installs.

### 2.4 Comparison Table

| Criterion | K-Means | HDBSCAN | BERTopic |
|---|---|---|---|
| Speed (~180 reviews) | ~10 ms | ~50 ms | ~1–2 s (UMAP + HDBSCAN + c-TF-IDF) |
| Interpretability | Low — centroid vectors only | Low — cluster IDs only | **High — c-TF-IDF keyword labels** |
| Handles noise | **No** — all reviews forced into a cluster | Yes — outliers labeled -1 | Yes — inherits HDBSCAN noise handling |
| Auto k | **No** — requires elbow + silhouette | Yes | Yes |
| Readable labels without LLM | No | No | **Yes** (c-TF-IDF) |
| Runs locally | Yes | Yes | Yes |
| Free | Yes | Yes | Yes |
| GPU required | No | No | No |
| TypeScript native | Yes (`ml-kmeans`) | No (Python) | No (Python) |
| Dimensionality reduction needed | Optional | Recommended (UMAP) | Built-in (UMAP) |

### 2.5 Recommendation: BERTopic

**Selected: BERTopic approach (UMAP + HDBSCAN + c-TF-IDF) with pre-computed Gemini embeddings.**

**Justification:**

1. **Auto k eliminates elbow/silhouette complexity.** Theme count varies week to week as product issues shift. Hardcoding k or running an optimization loop every week is fragile.
2. **Noise handling removes irrelevant reviews.** Generic reviews like "Good app" or "Nice" do not belong to any complaint theme. HDBSCAN's noise label (-1) excludes them, improving theme quality. K-Means would force these into a cluster, diluting theme coherence.
3. **c-TF-IDF reduces LLM dependency.** BERTopic generates interpretable keyword labels (e.g., "nominee, update, stuck, process") without any LLM call. The LLM only refines these into polished names (e.g., "Nominee Updates"), reducing the label generation prompt to a simple mapping task.
4. **No GPU, no PyTorch.** By supplying pre-computed Gemini embeddings, the pipeline avoids sentence-transformers and PyTorch. Dependencies: `bertopic`, `umap-learn`, `hdbscan`, `scikit-learn`, `numpy` — all lightweight, CPU-only.
5. **Batch-only execution.** The clustering pipeline runs weekly in GitHub Actions (or via manual trigger), not at request time. The Python dependency is isolated to the batch job and never enters the TypeScript application runtime.

**Implementation:** Python script (`scripts/cluster_reviews.py`) invoked from the GitHub Actions weekly workflow. For Admin manual triggers, the Next.js API route dispatches a GitHub Actions workflow via the GitHub API, or invokes the Python script via `child_process.exec`.

> **Cost flag: FREE.** All components (UMAP, HDBSCAN, c-TF-IDF, scikit-learn) run locally with zero API cost.

**Python dependencies (no GPU, no PyTorch):**

```
bertopic>=0.16
umap-learn>=0.5
hdbscan>=0.8
scikit-learn>=1.3
numpy>=1.24
```

## 3. Pipeline

### 3.1 Input

Masked reviews from the rolling 12-week window, read from the `reviews` table in the App DB.

```
SELECT id, review_text, rating, review_date
FROM reviews
WHERE review_date >= CURRENT_DATE - INTERVAL '12 weeks'
  AND review_text IS NOT NULL
ORDER BY review_date DESC
```

Typical volume: ~120–250 reviews over 12 weeks.

### 3.2 Pipeline Steps

```
Masked Reviews (rolling 12 weeks from DB)
    ↓
Step 1: Verify PII masking
    ↓
Step 2: Embed reviews (Gemini text-embedding-004, cache in DB)
    ↓
Step 3: Reduce dimensions (UMAP: 768 → 5)
    ↓
Step 4: Cluster (HDBSCAN: auto k, noise → -1)
    ↓
Step 5: Label clusters (c-TF-IDF → keyword labels)
    ↓
Step 6: Refine labels (cheapest LLM, 1 batched call)
    ↓
Step 7: Map to taxonomy (predefined or emergent)
    ↓
Step 8: Rank themes (by cluster size)
    ↓
Step 9: Select top 5 themes, top 3 for quotes
    ↓
Step 10: Extract 3 quotes per top-3 theme (nearest centroid, no LLM)
    ↓
Step 11: Generate action ideas (cheapest LLM, 1 call)
    ↓
Step 12: Generate pulse summary (cheapest LLM, 1 call, ≤250 words)
    ↓
Step 13: Assemble Review Pulse (matches spec 7.8)
    ↓
Step 14: Store in review_pulse table (cached, read by Scheduler + Dashboard)
```

### 3.3 Pipeline Pseudocode

```
FUNCTION weeklyThemeClassification():

  // ── Step 1: Fetch and verify PII masking ───────────────────
  reviews = DB.query(
    "SELECT * FROM reviews WHERE review_date >= NOW() - 12 weeks"
  )
  reviews = verifyPIIMasked(reviews)
  //   Re-mask any review_text containing PII patterns
  //   (should already be masked at ingestion — this is a safety net)

  // ── Step 2: Embed reviews ──────────────────────────────────
  FOR EACH review IN reviews:
    IF review has no cached embedding in review_embeddings table:
      embedding = embedWithGemini(review.review_text)
      //   Model: Gemini text-embedding-004 (free tier)
      //   Output: 768-dim vector
      DB.insert("review_embeddings", {
        review_id: review.id,
        embedding: embedding,
        model: "text-embedding-004"
      })

  embeddings = loadAllEmbeddings(reviews)
  //   Matrix shape: (N, 768) where N = number of reviews

  // ── Step 3: Dimensionality reduction ───────────────────────
  reduced = UMAP(
    n_components = 5,
    n_neighbors = 15,
    min_dist = 0.0,
    metric = "cosine"
  ).fit_transform(embeddings)
  //   Output shape: (N, 5)
  //   Cost: FREE (local algorithm)

  // ── Step 4: Cluster ────────────────────────────────────────
  cluster_labels = HDBSCAN(
    min_cluster_size = 5,
    min_samples = 3,
    metric = "euclidean"
  ).fit_predict(reduced)
  //   Output: array of integers, -1 = noise
  //   Cost: FREE (local algorithm)

  num_clusters = count_unique(cluster_labels, exclude = -1)
  noise_count = count(cluster_labels, where = -1)

  // ── Step 5: Generate c-TF-IDF labels ───────────────────────
  //   Group review texts by cluster ID
  //   Compute TF-IDF per cluster (c-TF-IDF)
  //   Extract top 5 keywords per cluster
  cluster_keywords = computeCTFIDF(reviews, cluster_labels)
  //   Example output:
  //   { 0: ["nominee", "update", "stuck", "change", "process"],
  //     1: ["login", "OTP", "failed", "logout", "password"],
  //     2: ["statement", "download", "tax", "capital", "gains"] }
  //   Cost: FREE (local TF-IDF)

  // ── Step 6: Refine labels with LLM (1 batched call) ────────
  predefined_categories = loadPredefinedTaxonomy()
  //   See Section 5 for full list

  refined_labels = LLM.call({
    model: "gemini-2.5-flash",
    prompt: LABEL_REFINEMENT_PROMPT(cluster_keywords, predefined_categories)
  })
  //   Input: all cluster keyword sets + predefined category list
  //   Output: { cluster_id → { label, is_predefined, predefined_match } }
  //   Cost: FREE (Gemini Flash free tier, ~200 tokens)

  // ── Step 7: Map to taxonomy ────────────────────────────────
  FOR EACH cluster IN refined_labels:
    IF cluster.is_predefined:
      cluster.canonical_name = cluster.predefined_match
      cluster.theme_type = "predefined"
    ELSE:
      cluster.canonical_name = cluster.label
      cluster.theme_type = "emergent"

  // ── Step 8: Rank by cluster size ───────────────────────────
  themes = []
  FOR EACH cluster_id IN unique(cluster_labels, exclude = -1):
    themes.append({
      label: refined_labels[cluster_id].canonical_name,
      theme_type: refined_labels[cluster_id].theme_type,
      count: count(cluster_labels == cluster_id),
      member_indices: indices(cluster_labels == cluster_id)
    })
  themes.sort(by = count, descending)

  // ── Step 9: Select top 5 / top 3 ──────────────────────────
  top5 = themes[0:5]
  top3 = themes[0:3]

  // ── Step 10: Extract 3 quotes per top-3 theme ──────────────
  //   Algorithmic: nearest to cluster centroid. No LLM call.
  FOR EACH theme IN top3:
    centroid = mean(embeddings[theme.member_indices], axis = 0)
    distances = cosine_distance(
      embeddings[theme.member_indices], centroid
    )
    nearest_3_indices = argsort(distances)[0:3]
    theme.quotes = reviews[theme.member_indices[nearest_3_indices]]
      .map(r => r.review_text)
      .map(truncate_to_120_chars)

  // ── Step 11: Generate action ideas (1 LLM call) ────────────
  action_ideas = LLM.call({
    model: "gemini-2.5-flash",
    prompt: ACTION_IDEAS_PROMPT(top3)
  })
  //   Input: top 3 themes with labels and quotes
  //   Output: exactly 3 action idea strings
  //   Cost: FREE (Gemini Flash free tier, ~150 tokens)

  // ── Step 12: Generate pulse summary (1 LLM call) ───────────
  pulse_summary = LLM.call({
    model: "gemini-2.5-flash",
    prompt: PULSE_SUMMARY_PROMPT(top5, reviews, ratings)
  })
  //   Input: top 5 themes, review volume, average rating, date range
  //   Output: summary string, max 250 words
  //   Cost: FREE (Gemini Flash free tier, ~350 tokens)

  // ── Step 13: Assemble Review Pulse ─────────────────────────
  pulse = {
    period: "Rolling 12 weeks ending {today}",
    product: "INDmoney",
    total_reviews_analyzed: reviews.length,
    average_rating: mean(reviews.map(r => r.rating)),
    top_themes: top5.map((t, i) => ({
      theme: t.label,
      rank: i + 1,
      quotes: t.quotes OR []    // only top 3 have quotes
    })),
    weekly_summary: pulse_summary,
    action_ideas: action_ideas,  // exactly 3 items
    top_customer_themes: top3.map(t => t.label),
    source: "Google Play Store Reviews"
  }

  // ── Step 14: Store in DB (cached) ──────────────────────────
  DB.insert("review_pulse", pulse)
  //   Also store per-theme trend data for Module B dashboard
  DB.insert("theme_snapshots", buildThemeSnapshots(themes, today))

  RETURN pulse

END FUNCTION
```

## 4. LLM Usage

### 4.1 Design Principle

Minimize LLM calls. Use the cheapest model. Prefer algorithmic approaches over LLM where possible.

### 4.2 Per-Task Breakdown

| Task | Method | Model | LLM Calls | Notes |
|---|---|---|---|---|
| Review embedding | Gemini Embedding API (batched) | text-embedding-004 | 0 (embedding, not LLM) | ~1 API call for new reviews (batched). Cached per review. |
| Dimensionality reduction | UMAP (local) | N/A | 0 | Runs locally, no API |
| Clustering | HDBSCAN (local) | N/A | 0 | Runs locally, no API |
| c-TF-IDF labels | scikit-learn (local) | N/A | 0 | Runs locally, no API |
| **Label refinement** | LLM (batched) | **Gemini 2.5 Flash** | **1** | All cluster labels in one prompt |
| Quote selection | Nearest centroid (local) | N/A | **0** | Cosine distance, no LLM |
| **Action ideas** | LLM | **Gemini 2.5 Flash** | **1** | One call, all 3 ideas |
| **Pulse summary** | LLM | **Gemini 2.5 Flash** | **1** | One call, ≤250 words |

### 4.3 Total LLM Calls Per Weekly Run

| Call Type | Count | Model | Estimated Tokens |
|---|---|---|---|
| Label refinement | 1 | Gemini 2.5 Flash | ~200 |
| Action ideas | 1 | Gemini 2.5 Flash | ~150 |
| Pulse summary | 1 | Gemini 2.5 Flash | ~350 |
| **Total LLM calls** | **3** | | **~700 tokens** |

Additionally, **1 embedding API call** (batched) for new reviews using Gemini text-embedding-004. This is not an LLM call.

At 3 LLM calls per weekly run, this is far within Gemini 2.5 Flash free tier limits (10 RPM, 500 RPD).

## 5. Theme Taxonomy

### 5.1 Predefined Categories

Based on INDmoney's known issue landscape from Google Play reviews:

| # | Predefined Theme | Typical Keywords |
|---|---|---|
| 1 | Nominee Updates | nominee, update, change, beneficiary |
| 2 | Login Issues | login, OTP, password, authentication, logout |
| 3 | Statement Downloads | statement, download, tax, capital gains, report |
| 4 | SIP / Mandate Issues | SIP, mandate, auto-debit, recurring, payment failed |
| 5 | Withdrawal Delays | withdrawal, redeem, delay, pending, money stuck |
| 6 | Fee / Charge Confusion | fee, charge, deducted, expense ratio, exit load |
| 7 | Account Changes | account, deactivate, close, transfer, migrate |
| 8 | App Performance | crash, slow, hang, freeze, lag, unresponsive |
| 9 | KYC Issues | KYC, verification, Aadhaar, PAN, document, rejected |
| 10 | Customer Support | support, response, helpline, no reply, ticket |

### 5.2 Mapping Logic

After c-TF-IDF generates keyword labels for each cluster, the LLM label refinement call (Section 4.4) maps each cluster to either a predefined category or an emergent label.

```
FOR EACH cluster:
  IF LLM maps cluster to a predefined category:
    cluster.canonical_name = predefined category name (exact string)
    cluster.theme_type = "predefined"
  ELSE:
    cluster.canonical_name = LLM-generated emergent label
    cluster.theme_type = "emergent"
```

### 5.3 When to Surface as Emergent

A theme is labeled **emergent** when:

1. The c-TF-IDF keywords do not semantically match any predefined category.
2. The LLM label refinement confirms no predefined match (`is_predefined: false`).
3. The cluster has `min_cluster_size` or more reviews (not noise).

Emergent themes appear in the Review Pulse with their LLM-generated label and are flagged as `"theme_type": "emergent"` in the `theme_snapshots` table so the Admin dashboard can highlight them.

### 5.4 Taxonomy Interface

```typescript
interface ThemeLabel {
  cluster_id: number;
  label: string;
  theme_type: "predefined" | "emergent";
  predefined_match: string | null;
  review_count: number;
  keywords: string[];
}
```

## 6. Week-over-Week Trends

### 6.1 Trend Computation

Each week, after clustering, compare the current week's theme distribution against the previous week's stored `theme_snapshots`.

```
FOR EACH theme IN current_week_themes:
  previous = lookupPreviousWeek(theme.canonical_name)

  IF previous IS NULL:
    theme.trend_status = "emerging"
    theme.wow_change_percent = null

  ELSE:
    wow_change = ((theme.count - previous.count) / previous.count) * 100

    IF wow_change > +20:
      theme.trend_status = "worsening"
    ELSE IF wow_change < -20:
      theme.trend_status = "improving"
    ELSE:
      theme.trend_status = "stable"

    theme.wow_change_percent = wow_change
```

### 6.2 Trend Thresholds

| Condition | Trend Status | Dashboard Signal |
|---|---|---|
| Theme not in previous week | `emerging` | New theme badge |
| wow_change_percent > +20% | `worsening` | Red arrow up |
| wow_change_percent < -20% | `improving` | Green arrow down |
| -20% ≤ wow_change_percent ≤ +20% | `stable` | Gray dash |

### 6.3 Sentiment Derivation

Sentiment distribution (per spec section 8.2) is derived from review ratings without LLM:

| Rating | Sentiment |
|---|---|
| 1–2 | Negative |
| 3 | Neutral |
| 4–5 | Positive |

This avoids an LLM call for sentiment classification.

### 6.4 Theme Matching Across Weeks

Stable theme names are critical for week-over-week comparison. The predefined taxonomy (Section 5) ensures consistent names across weeks:

- **Predefined themes** always use the exact predefined name. "Login Issues" this week and "Login Issues" last week compare directly.
- **Emergent themes** are matched by label string. If the same emergent label appears in consecutive weeks, it is compared. If it does not recur, it drops off the trend view.

### 6.5 Trend Output Interface

```typescript
interface ThemeTrend {
  theme: string;
  current_week_count: number;

}

interface WeeklyTrendReport {
```

## 7. Review Pulse Generation

### 7.1 Caching Strategy (Per ADR-010)

The Review Pulse is **cached in the `review_pulse` DB table** and regenerated weekly after each ingestion run.

| Property | Value |
|---|---|
| Storage | `review_pulse` table in App DB |
| Regeneration trigger | Weekly, after GitHub Actions ingestion run completes |
| Staleness threshold | >7 days (per edgeCase.md) |
| Stale behavior | Trigger regeneration before serving, show loading state |
| Manual trigger | Admin dashboard button dispatches GitHub Actions workflow |

### 7.2 Generation Trigger Sequence

```
GitHub Actions weekly cron (or Admin manual trigger)
    ↓
google-play-scraper fetches new reviews
    ↓
Store reviews in DB (dedup, PII mask)
    ↓
Run weeklyThemeClassification() (Section 3.3)
    ↓
Review Pulse stored in review_pulse table
    ↓
Theme snapshots stored in theme_snapshots table
    ↓
Dashboard reads from DB (no LLM call)
```

### 7.3 Review Pulse Schema

Stored in the `review_pulse` table (defined in architecture.md Section 5.1):

```json
{
  "id": "uuid",
  "product": "INDmoney",
  "period": "Rolling 12 weeks ending 2026-04-26",
  "total_reviews_analyzed": 180,
  "average_rating": 3.2,
  "top_themes": [
    {
      "theme": "Nominee Updates",
      "rank": 1,
      "quotes": ["...", "...", "..."]
    }
  ],
  "weekly_summary": "This week, Google Play reviews continue to show...",
  "action_ideas": ["...", "...", "..."],
```

The stored pulse must contain exactly 5 `top_themes`, exactly 3 quotes for each of the top 3 themes, empty `quotes` arrays for ranks 4-5, and exactly 3 `action_ideas`.

### 7.4 Edge Cases (Per edgeCase.md)

| Scenario | Behavior |
|---|---|
| Scraper returns 0 reviews | Log as `partial_success`, do not overwrite previous pulse |
| Fewer than `min_cluster_size` reviews total | Skip clustering, return previous cached pulse |
| All reviews assigned to noise (no clusters) | Lower `min_cluster_size` to 3, retry once. If still no clusters, return previous pulse with a flag. |
| Cached pulse is stale (>7 days) | Trigger regeneration before serving, show loading state |
| LLM rate limit hit during generation | Queue and retry with exponential backoff (per edgeCase.md) |

## 8. Top Themes Handoff to Scheduler

### 8.1 What the Scheduler Needs

The Advisor Scheduler (Module D) reads the latest top customer themes at greeting time to make the customer-facing greeting context-aware (per architecture.md Section 3.1).

The Scheduler needs:
- Theme names (human-readable strings)
- No embeddings, no cluster data, no quotes

### 8.2 Storage

The `top_customer_themes` field in the `review_pulse` table stores the top 3 theme names as a JSON array of strings:

```json
["Nominee Updates", "Login Issues", "Statement Downloads"]
```

### 8.3 How the Scheduler Reads (DB Read, Not LLM Call)

The Scheduler performs a single DB query at greeting time:

```sql
SELECT top_customer_themes
FROM review_pulse
ORDER BY created_at DESC
LIMIT 1
```

```typescript
interface SchedulerThemeRead {
  top_customer_themes: string[];
}

async function getLatestTopThemes(): Promise<string[]> {
  const result = await db.query(
    "SELECT top_customer_themes FROM review_pulse ORDER BY created_at DESC LIMIT 1"
  );
  if (!result || result.length === 0) return [];
  return JSON.parse(result[0].top_customer_themes);
}
```

**No LLM call.** The Scheduler reads pre-computed theme names from the database. The themes were generated during the weekly clustering pipeline and cached.

### 8.4 Scheduler Greeting Integration

The Scheduler uses the same canonical greeting template defined in `voiceAgent.md` for both chat and voice:


## 9. PII Masking Before Clustering

### 9.1 Masking Checkpoint

PII masking occurs at **two** points before review text reaches the clustering pipeline:

| Checkpoint | When | What |
|---|---|---|
| **Primary: Ingestion time** | Module A, after `google-play-scraper` returns raw reviews | Replace PII patterns with `[REDACTED]` before storage in `reviews` table. No raw PII is ever written to the DB. |
| **Secondary: Pre-clustering verification** | Step 1 of the clustering pipeline (Section 3.3) | Re-scan all reviews in the 12-week window for PII patterns before embedding. Safety net for any masking gaps. |

### 9.2 PII Patterns Masked

Per rules.md PII Rules:

| PII Type | Pattern | Replacement |
|---|---|---|
| Phone number | `\b\d{10}\b`, `\+91\d{10}` | `[REDACTED]` |
| Email | `\b[\w.-]+@[\w.-]+\.\w+\b` | `[REDACTED]` |
| PAN | `\b[A-Z]{5}\d{4}[A-Z]\b` | `[REDACTED]` |
| Aadhaar | `\b\d{4}\s?\d{4}\s?\d{4}\b` | `[REDACTED]` |
| Account number | `\b\d{9,18}\b` (in financial context) | `[REDACTED]` |
| Full name (in signature patterns) | Heuristic detection | `[REDACTED]` |

### 9.3 Why Mask Before Embedding

If PII is present in review text when it is embedded:
1. The embedding vector encodes PII-related semantic content.
2. The cluster centroid inherits PII signal.
3. Nearest-centroid quote selection might prefer PII-containing reviews.
4. PII could leak into the Review Pulse output.

Masking before embedding ensures the entire clustering pipeline — embeddings, centroids, quotes, labels — is PII-free.
